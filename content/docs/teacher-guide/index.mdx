## 最初に受講者へ伝えること

- 「今日は90分で、ゲームを1本完成させます」
- 「全部を暗記しなくて大丈夫です。まずは見本どおりに完成させましょう」
- 「困ったら、今どのステップで止まっているかを教えてください」

## 90分進行台本

| 時間 | やること | 教員の焦点 | 遅れたときの対応 |
| --- | --- | --- | --- |
| 0～10分 | 「はじめに」ページを開いてデモ動画を再生 → 「今日の流れ」ページでタイムライン図を説明する | デモ動画で完成形を見せてから各フェーズの目的を一言ずつ説明する | 駆け足でよいので両ページとも必ず通す |
| 10～25分 | Third Person プロジェクト作成 | 保存場所とテンプレート選択を全員そろえる | 個別対応より画面差分の共通確認を優先 |
| 25～40分 | BP_Item 作成と配置 | Blueprint 化の意味を一言で伝える | 配置数は3個でもよい |
| 40～65分 | Overlap と Score 加算 | Destroy Actor より前に加算する順番を徹底する | 見た目の調整は省略 |
| 65～80分 | BP_Goal とクリア条件 | Score >= RequiredScore を全員で確認する | Widget が間に合わなければ Print String で代替可 |
| 80～90分 | 通しプレイ、振り返り | 「自分が作った仕組み」を言葉にさせる | 発表は1～2名で十分 |

## 詰まったときの声かけ

- 「まずは完璧より、講師画面と同じ状態まで持っていきましょう」
- 「今は意味を全部覚えなくて大丈夫です。つながると何が起きるかを見ましょう」
- 「止まっている人は、今開いている画面がレベルか Blueprint かだけ確認してください」
- 「時間が足りなくなったら、見た目の調整は後回しで大丈夫です」
- 「完成した人は、ゲームを作ろうページの発展演習に進んでください」

## クロージング

「今日やったのは、ゲーム開発のほんの入口です。
でも、当たり判定・変数・条件分岐をつなぐと、ゲームがちゃんと動くことを自分の手で確かめられました。
ここから先は、ルールを1つ増やすたびに、作れるものが広がっていきます。」
