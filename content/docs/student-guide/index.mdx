ここから実際に手を動かしてゲームを作っていきます。

このページの手順をひとつずつ進めれば、最後にはキャラクターが動いてアイテムを集めてクリアになるゲームが完成します。何か困ったら、すぐにサポートしに行きますので、お気軽に手を挙げて教えてください。

<Section title="2-3. キューブを Blueprint クラスに変換する" goal="キューブが BP_Item という Blueprint クラスになります">

<Concept title="Blueprint クラスとは">

Unreal Engine でオブジェクトに「触れたら消える」「最後の 1 個ならクリア表示を出す」といった動作を設定するには、**Blueprint クラス**にする必要があります。

Word や Google スライドの「テンプレート」を想像してください。テンプレートを 1 つ作れば、同じデザインのコピーを何個でも作れます。Blueprint クラスも同じで、1 個設定すれば 5 個に増やせます。

</Concept>

<Concept title="「BP_」プレフィックスの意味">

この後のダイアログでは、Blueprint の名前を **BP_Item** にします。ファイル名の先頭に「BP_」を付けるのは、一覧を見たときに「これは **Blueprint** だ」とすぐわかるようにするための慣習です。

</Concept>

<Concept title="詳細パネルとは">
ビューポートなどで選択中のオブジェクトの設定項目（プロパティ）が一覧表示されるパネルです。通常は画面右側にあり、ここで選んだものの見た目や動作を細かく変更していきます。
</Concept>

<Action img="./img/convert-to-blueprint-process.png" alt="ビューポートでキューブを選択し、詳細パネルにブループリントクラスへの変換ボタンが表示されている">
  ビューポートでキューブを**クリックして選択**し、詳細パネルの「**ブループリントクラスへの変換**」ボタンを押します
</Action>

<Action img="./img/convert-to-blueprint-window-process.png" alt="Blueprint クラス作成ダイアログ。①親クラス ②パス ③ブループリント名入力欄 ④アセットパス ⑤選択ボタン">
  ①②④ は画像の通りになっているか見比べて、違っていれば画像と同じ状態にします（最初から画像の通りになっているはずです）。③ のブループリント名に「**BP_Item**」と入力して、⑤「**選択**」をクリックします
</Action>

<Verify img="./img/after-convert-to-blueprint.png" alt="コンテンツブラウザに BP_Item が追加され、Blueprint 編集画面が自動で開いた状態">コンテンツブラウザに `BP_Item` が表示されて Blueprint 編集画面が自動で開けば変換成功です</Verify>

<Action img="./img/close-BP_Item-editor-window.png" alt="BP_Item 編集画面のタブ。閉じるボタンが示されている">
  自動的に開いた `BP_Item` の編集画面は今は使わないので、タブを閉じます

<Section title="Step 3：ゲームを完成させる（25〜60分）" goal="アイテムに触れると消え、5 個すべて集めた瞬間に「CLEAR!」が表示されるゲームが完成します">

ここからは、**触れたことを検知する** → **最後の 1 個かを判定する** という順番で、ゲームの仕組みを理解しながら組み立てていきます。

| 仕組み | 片方だけ設定するとどうなる？ |
| --- | --- |
| **触れたことを検知する** | 何も起きない |
| **最後の 1 個か判定する** | 最後まで集めても CLEAR! が出ない |

<Section title="3-1. BP_Item のコリジョン設定を変える" goal="プレイヤーが BP_Item をすり抜けて通れるようになります">

<Concept title="コリジョンとは">

**コリジョン（Collision）** とは、オブジェクト同士が「ぶつかった・触れた」ことをゲームが認識するための当たり判定のことです。

| 種類 | 動作 | 使いどころ |
| --- | --- | --- |
| **ブロック** | 相手をはじいて通れなくする | 壁・床・障害物 |
| **オーバーラップ** | すり抜けられる＋触れたことだけ通知する | アイテム・ゴールゾーン |

BP_Item は元キューブなので初期設定は「ブロック」です。「オーバーラップ」に変えることで、すり抜けつつ触れた瞬間を検知できるようになります。

</Concept>

<Action>
  コンテンツブラウザで「**BP_Item**」をダブルクリックして編集画面を開きます
</Action>

<Action img="./img/how-to-BP_Item_OverlapAllDynamic.png" alt="BP_Item の詳細パネル。①コリジョン設定 ②Generate Overlap Events ③コリジョンプリセット OverlapAllDynamic が示されている">
  ①「**コリジョン**」で検索して、②「**Generate Overlap Events**」をオン、③「**コリジョンプリセット**」を ④「**OverlapAllDynamic**」にします
</Action>

<Verify img="./img/BP_Item_GenerateOverlapEvents_and_OverlapAllDynamic.png" alt="BP_Item の詳細パネル。Generate Overlap Events がオン、コリジョンプリセットが OverlapAllDynamic に切り替わっている">**Generate Overlap Events** がオン、**コリジョンプリセット** が **OverlapAllDynamic** に切り替わっていれば設定完了です</Verify>

</Section>

<Action img="./img/open-full-blueprint-editor-button.png" alt="BP_Item 編集画面。フルブループリントエディタを開くボタンがハイライトされている">

<Concept title="イベントグラフとは">

![BP_Item のイベントグラフ](./img/BP_Item-EventGraph.png)

ここにノードを配置して、**触れたらどう動くか**を組み立てていきます。最初からある `Event ActorBeginOverlap` が「何かに触れた瞬間」のスタート地点です。

</Concept>

<Reference title="グラフ操作で困ったとき">

<Concept title="Event ActorBeginOverlap と Print String">

- **Event ActorBeginOverlap**：「何かが触れた瞬間」に起動するノードです。
- **Print String**：指定した文字を画面に表示するノードで、仕組みが正しく動いているかを手軽に確認するために使います。

</Concept>

<Action img="./img/add-print-string-node-from-actorbeginoverlap.png" alt="Event ActorBeginOverlap の実行ピンから Print String を検索して追加する手順">
  `Event ActorBeginOverlap` の**実行ピン**からドラッグして、「**Print String**」を追加します
</Action>

<Action>
  `Print String` の **In String** 欄に「**Hello**」と入力します
</Action>

<Concept title="「コンパイル」とは">

Blueprint のノードを組み終わったら、**コンパイル**でゲームに反映します。文書を「保存」するのに近い操作ですが、正確には「作ったノードの設定をゲームが実際に動かせる形に変換する」作業です。コンパイルしないと、ノードを変更してもゲームには反映されません。

ツールバーのコンパイルボタンのアイコンで結果がわかります。

| アイコン | 意味 |
| --- | --- |
| **緑のチェック** | 成功。ゲームに反映されました |
| **赤の ×** | エラーあり。修正が必要です |

ノードを組み終わったら一度押してアイコンを確認する習慣をつけておくと、問題を早く発見できます。

</Concept>

<Action img="./img/compile-BP_Item-and-check-result.png" alt="Blueprint エディタのツールバー。コンパイルボタンがハイライトされている">
  「**コンパイル**」をクリックします

</Recovery>

</Action>

<Section title="3-3. 最後の 1 個を判定するロジックを組む" goal="アイテムが消え、最後の 1 個を取ったときだけ「CLEAR!」が表示されるようになります">

<Concept title="Hello チェックで学んだ Blueprint の基本の流れ">

![BP_Item が何かと触れた時に、画面に「Hello」と表示](./img/print-string-when-Event%20ActorBeginOverlap-explain.png)

**Event ActorBeginOverlap** が「触れた瞬間」を起点にして処理を起動し、そこにつないだ **Print String** が「Hello」を画面に表示した、という流れです。「何かが起きたら → つないだ先の処理が順番に動いていく」というのが Blueprint の基本です。ここから先は Print String の代わりに「アイテムを消す」「最後の 1 個なら CLEAR! を出す」処理をつないで本番のゲームにしていきます。

</Concept>

<Concept title="Destroy Actor / Get All Actors of Class / Length / == 0 / Branch の役割">

本番ロジックで使う 5 つのノードの役割をまとめます。

| ノード | 何をするか |
| --- | --- |
| **Destroy Actor** | このアイテム自身をレベルから消す |
| **Get All Actors of Class** | レベル上にある `BP_Item` を全部リストアップする |
| **Length** | そのリストが何個あるかを数えて数値で返す |
| **== 0** | 数値が 0 かどうかを比べて True / False を返す |
| **Branch** | Condition が True なら True 側へ、False なら False 側へ処理を分ける |

`Destroy Actor` で自分を消した**あと**に `Get All Actors of Class` で残りを数えることで、「残りが 0 個 = 全部集め終わった」と判定できます。残りが 0 個なら `== 0` が True になり、`Branch` の True 側に進んで `CLEAR!` が表示されます。

</Concept>

<Action>
  さきほど追加した `Print String` ノードを選択して **Delete** キーで削除します
</Action>

<Action img="./img/add-destroy-actor-node.png" alt="Event ActorBeginOverlap の実行ピンの後ろに Destroy Actor ノードが追加された状態">
  `Event ActorBeginOverlap` の後ろに「**Destroy Actor**」を追加します
</Action>

</Action>

<Action img="./img/add-branch-after-destroy-actor.png" alt="Destroy Actor の後ろに Branch ノードが追加された状態">
  `Destroy Actor` の後ろに「**Branch**」を追加し、さきほど作った `==` ノードの出力を `Branch` の `Condition` へつなぎます
</Action>

<Action img="./img/add-clear-message-print-string.png" alt="Branch の True 実行ピンから Print String が配置され、In String に CLEAR! と入力されている">
  `Branch` の `True` の後ろに「**Print String**」を追加して、「**CLEAR!**」と入力します
</Action>

<Action>

</Action>

<Verify>最後の 1 個ではない段階では、アイテムは消えても「CLEAR!」が表示されなければ成功です。確認できたら Esc でプレイを終了します。</Verify>

<Action img="./img/compile-BP_Item-and-check-result.png" alt="Blueprint エディタのツールバー。コンパイルボタンがハイライトされている">
  「**コンパイル**」をクリックします
</Action>

<Recovery title="コンパイルアイコンが赤い × のとき">
コンパイルエラーです。画面下部の「出力ログ」パネルにエラーメッセージが表示されます。エラー行をダブルクリックすると問題のノードにジャンプできます。修正したら、もう一度「コンパイル」を押してください。
</Recovery>

<Reference title="イベントグラフの流れ">

`Event ActorBeginOverlap` → `Destroy Actor` → `Get All Actors of Class(BP_Item)` → `Length` → `== 0` → `Branch` → `True` のときだけ `Print String(CLEAR!)`

</Reference>

</Section>

<Section title="3-4. テストプレイで確認する" goal="最後の 1 個を取った瞬間に「CLEAR!」が表示されるゲームが完成します">

<Action>

## 振り返り（50〜60分）

今日作ったゲームには、3つのプログラミングの基本的な仕組みが使われています。

| 仕組み | 意味 |
| --- | --- |
| **当たり判定** | 「触れた」ことをゲームが認識する |

## 発展演習：ゲームをカスタマイズしよう（60〜90分）

ゲームが完成したら、余った時間で挑戦してみましょう。

### 演習-発展1：アイテムをくるくる回す

<Exercise>

1. `BP_Item` を開き、コンポーネントパネルの「**＋ 追加**」をクリック → 検索欄に「**回転**」と入力 →「**回転中移動コンポーネント**」を選んで追加します。
2. コンポーネントパネルで「**Static Mesh Component**」を選択 → 詳細パネルのトランスフォームカテゴリの「**可動性**」を「**ムーバブル**」に変更します。
3. コンパイル・保存してプレイします。

→ アイテムがくるくる回っていれば成功です。

</Exercise>

### 演習-発展2：アイテムを専用モデルに差し替える

<Exercise>

無料・会員登録なしで `.fbx` を探せるサイトの例です。好きなサイトを 1 つ開き、好きな 3D 素材を 1 つダウンロードしましょう。

- [Kenney](https://kenney.nl/assets/category:3D)
- [Poly Haven](https://polyhaven.com/)
- [NASA 3D Resources](https://science.nasa.gov/3d-resources/)
- [Open3dModel](https://open3dmodel.com/)

ダウンロードした zip を展開して、中にある好きな `.fbx` を 1 つ選んでください。

選んだ `.fbx` を Unreal Editor の **コンテンツブラウザ** 内の **コンテンツ** 以下の自分のわかりやすい場所にドラッグ＆ドロップします。インポートダイアログが開いたら「**インポート**」をクリック。

次に `BP_Item` を開き、コンポーネントパネルで「**Static Mesh Component**」を選択 → 詳細パネルのスタティックメッシュカテゴリの「**Static Mesh**」を、今インポートしたモデルに変え、マテリアルカテゴリにインポートしたマテリアルを割り当て、コンパイル・保存してプレイ。

→ 選んだモデルのアイテムが置いた場所に表示されていれば成功です。

</Exercise>

### 演習-発展3：クリアしたらスローモーションにする

<Exercise>

`BP_Item` のイベントグラフで、`Print String`（`CLEAR!` を表示するもの）の実行ピンの後ろに「**Set Global Time Dilation**」ノードをつなぎ、値を `0.3` にしてコンパイル・保存してプレイ。

→ アイテムに触れた瞬間に動きがゆっくりになれば成功です。

</Exercise>
